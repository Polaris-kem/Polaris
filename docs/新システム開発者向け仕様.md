# 新システム開発者向け仕様書（ERP基幹システム）

営業から製造・出荷・経理までを統合管理する基幹システム（ERP）の再構築において、設計・実装に必要な情報をソースに基づきまとめたドキュメントです。既存の `supabase-table-schemas.sql` および `docs/業務コンテキスト.md`・`docs/生産管理システム設計指針.md` と併せて参照してください。

---

## 1. システム全体概要とコア・ワークフロー

### 1.1 主キーとデータ紐付け

- **工事番号（受注番号）** と **受注登録日（KeyDate）** を**複合の論理主キー**として、全工程のデータを紐付ける。
- 工事番号は数年で再利用されるため、**必ず KeyDate（受注登録日）とセット**で一貫性を保つ。テーブル設計・画面・API では「工事番号のみ」での一意参照を避け、可能な限り `(constructno, keydate)` または `(constructno, registerdate)` で識別する。

### 1.2 基本フロー（部門別）

| 順序 | 部門 | 主な業務 |
|------|------|----------|
| 1 | **営業部** | 見積依頼（製造管理部へ）→ 見積書作成 → 受注登録（製造指図書発行）。 |
| 2 | **設計部門** | 部品リスト（工事企画表・購入部品表）の作成・出力。 |
| 3 | **製造管理部** | 設計データのインポート → 材料・外注加工・購入部品の発注管理。 |
| 4 | **製造現場（明石工場等）** | スマホアプリによる納入処理・加工完了登録・作業票（工数）登録。 |
| 5 | **経理・総務** | 請求書処理（買掛管理）・売上登録・入金管理。 |

## 2. データベース構造（主要テーブルとビュー）

テーブル名は**すべて小文字**で定義する。

### 2.1 主要マスタテーブル

| テーブル名 | 説明 | 備考 |
|------------|------|------|
| **t_companycode** | 取引先情報 | 売掛・買掛コード、支払条件、メールアドレス、税計算ルール等。 |
| **t_staffcode** | 社員情報 | 所属部署、ログイン権限等。 |
| **t_workcode** | 職種の定義コード | 作業内容の分類。 |
| **t_worktimecode** | 作業内容の定義コード | 工数入力時の選択肢。 |

### 2.2 業務プロセス用テーブル

| テーブル名 | 説明 | 備考 |
|------------|------|------|
| **t_acceptorder** | 受注基本情報 | 注文主、金額、納期、通貨等。`ConstructNo` + `RegisterDate`（KeyDate）で他テーブルと紐付け。 |
| **t_construction** | 工事基本情報 | 受注元と納入先の紐付け。 |
| **t_manufctparts** | 製作部品の進捗 | 完了予定日、完了日、図面番号等のステータス管理。 |
| **t_purchaseparts** | 購入部品 | 発注・検収実績。 |
| **t_manufctpurchase** | 外注加工・材料 | 発注・検収実績。 |
| **t_worktime** | 社内工数実績 | 機械単位または図面単位（後述の入力ルールに従う）。 |
| **t_worktimekako** | 社内工数実績（加工） | 加工部門用の工数データ。 |

### 2.3 集計ビュー（ロジックの参照元）

| ビュー名 | 説明 |
|----------|------|
| **v_加工進捗** | 部品単位の現在の状態を可視化。 |
| **v_工事番号別原価集計** | 夜間バッチで計算された原価・粗利を確認。 |

### 2.4 既存スキーマとの対応

- 現在の `supabase-table-schemas.sql` には **t_saiban, t_machinemarkforsaiban, t_computerdevice, t_acceptorder, t_staffcode** が定義済み。
- **t_companycode, t_construction, t_manufctparts, t_purchaseparts, t_manufctpurchase, t_worktime, t_worktimekako** および集計ビューは、`sql/erp-schema-reference.sql`（本リポジトリに追加）を参照してスキーマを拡張する。

## 3. 重要なビジネスロジックと運用ルール

開発者が実装すべき特有のロジックです。

### 3.1 工事番号の巡回性

- 工事番号は**数年で再利用**される。
- **必ず「受注登録日（KeyDate）」とセット**でデータの一意性・一貫性を保持する。
- 検索・更新・削除・API では `(constructno, keydate)` または `(constructno, registerdate)` を条件に含めることを推奨する。

### 3.2 工数入力ルール

| 項目 | 値 | 備考 |
|------|-----|------|
| 1日合計 | **7.75時間** | 定数としてプログラムに持つ。 |
| 入力単位 | **0.25時間（15分）** | 入力値はこの単位でバリデーション。 |
| 加工部門 | **図面番号単位**で登録 | 図面番号ごとに工数を記録。 |
| 組立・設計部門 | **機械（工事番号）単位**で登録 | 工事番号ごとに工数を記録。 |

### 3.3 発注管理の区分（注文番号プレフィックス）

| プレフィックス | 意味 | 用途 |
|----------------|------|------|
| **P番** | 購入部品 | 個別注文番号。 |
| **R番** | 材料 | 個別注文番号。 |
| **C番** | 外注加工 | 個別注文番号。 |
| **A番** | 一括 | 一括注文番号。 |

※ 既存の用語体系（P=工事、C=外注、A=購入、FY=経費）と併用する場合は、文脈で「工事のP」と「購入部品のP番」を区別する。発注番号のプレフィックスは上記 P/R/C/A を採用。

### 3.4 夜間バッチ処理

- 原価集計などの**重い計算は夜間に実行**し、集計テーブル・ビューに書き出す。
- 日中のパフォーマンスを維持するため、オンライン画面では集計結果を**参照のみ**とし、再計算はバッチに任せる。

### 3.5 電子帳簿保存法対応

- 受注・発注の各段階で、**見積書・注文書・請求書（PDF）のアップロード**とメタデータの紐付けを必須とする。
- 既存の「電子見積書・電子注文書・電子請求書」カラム（t_acceptorder 等）を活用し、帳簿として保存可能な形式で格納する。

## 4. 外部インターフェース・機能要件

### 4.1 Excel インポート

- 設計部門が出力した**パーツリスト（Excel）**を読み込む。
- **開始行（デフォルト8行目）**などのパラメータを設定可能にする。

### 4.2 自動メール送信

- **納期変更時の承認フロー**: `approvedeps` 配列に基づき関係者へ通知する。
- **業者向け**: 注文書・現品票の自動送信。

### 4.3 スマホ連携

- スマホカメラによる**バーコード読み取り**で以下を実施：
  - 納入処理
  - 加工完了登録

### 4.4 マルチ通貨

- 円貨に加え、**USドル等の外貨**での受注・発注に対応。
- **為替レート**の保持と計算を行う（t_acceptorder の Currency, ForgnMoney 等を活用）。

---

## 5. 主要なデータ構造（構造体例）

- **s_estimatedata**: 見積から受注へ引き継ぐ詳細情報。見積コピー＆マージン加算時に使用。
- **t_zumendata_manufct**: 図面DBからインポートされる製作部品の基本構造。

これらは既存の VB.NET 等のソースに定義がある場合は、そのフィールドを踏襲しつつ API やフロントの型定義に落とす。
