# 機能別画面遷移図とビジネスプロセスに沿ったデータフロー

Production Control システムの全体像を、**機能別画面遷移図（メニュー構造）**と**ビジネスプロセスに沿ったデータフロー（DFD）**としてまとめたリファレンスです。プログラム開発者が全体構造を把握し、モジュール間の依存関係を設計する際に利用してください。

**関連ドキュメント**: `docs/新システム開発者向け仕様.md`, `docs/画面と業務の対応.md`, `docs/業務コンテキスト.md`

---

## 1. 機能別画面遷移図（主要メニュー構造）

システムは**役割（ロール）**ごとにメニューが整理されています。

```
[メインメニュー]
│
├─ 経理／営業 モジュール
│   ├─ 受注関係画面
│   │   ├─ 見積依頼
│   │   ├─ 見積書作成
│   │   └─ 受注登録・製造指図書発行
│   ├─ 請求・売上画面
│   │   ├─ 和文/英文請求書発行
│   │   └─ 売上日登録
│   └─ 検索画面
│       ├─ 見積検索
│       └─ 図面番号から実績検索 等
│
├─ 発注部門 モジュール
│   ├─ データ取込画面
│   │   ├─ 図面DBからの取り込みと登録
│   │   ├─ 購入部品表
│   │   └─ 工事企画表
│   ├─ 発注管理画面
│   │   ├─ 発注データの選択
│   │   ├─ 個別/一括発注処理
│   │   └─ 電子文書アップロード
│   ├─ 諸経費管理画面
│   │   ├─ 諸経費発注
│   │   └─ 見積書・請求書登録
│   └─ 検収管理画面
│       └─ 全種類 請求書アップロード、金額照合
│
├─ 原価・進捗関係 モジュール
│   ├─ 進捗検索画面
│   │   └─ 加工進捗検索（工事番号/機械/ユニット単位）
│   ├─ 作業票登録画面
│   │   └─ 工数入力（機械単位/図面単位）
│   └─ 原価集計表示
│       └─ 夜間バッチ結果の閲覧（案件別/機械別/ユニット別）
│
└─ 製造・組立現場（モバイル/PC）
    ├─ スマホアプリ
    │   ├─ 納入処理
    │   ├─ 加工完了登録
    │   └─ リンマン登録
    └─ 工事番号出荷登録画面
```

### 1.1 既存 Polaris 画面との対応（参考）

| 機能分類 | 既存画面（index.html / showPage） |
|----------|-----------------------------------|
| 受注関係 | 受注登録（order-registration）、受注データ一覧（accept-order-list） |
| 検索 | 検索ページ（search）、加工進捗管理（processing-progress）、全テーブル検索（list） |
| データ取込・発注 | 購入部品納入処理（parts-delivery）、外注・材料納入処理（outsource-delivery）、諸掛購入登録（misc-purchase）、諸掛納入処理（misc-delivery） |
| 諸経費 | 諸掛購入登録、諸掛納入処理 |
| 作業票・工数 | 作業票の登録（work-ticket）、バーコードリーダー版作業票 |
| 出荷 | 出荷管理（shipping）、工事番号出荷登録（project-shipping） |
| 図面・採番 | 図面番号採番（drawing-number）、工事番号採番（construct-number） |
| 余剰・在庫 | 余り品登録（surplus-register）、余り品検索（surplus-search） |

※ 見積依頼・見積書作成・請求書発行・検収管理・原価集計表示・スマホアプリは、既存画面の拡張または新規モジュールとして実装する。

## 2. データフロー図（DFD）

業務の流れに沿った**データの発生・移動・保存先（テーブル）**を定義します。

### 2.1 全体フロー概要

```
[受注・設計] → [調達・製造] → [検収・売上]
     │               │               │
  見積・受注      発注・納入・工数    検収・集計・売上
```

---

### ① 受注・設計フェーズ

| ステップ | 内容 | 主な保存先・連携 |
|----------|------|-------------------|
| **1. 見積依頼** | 営業部が「見積依頼（IQ）」を作成。製造管理部へ通知。 | `t_inquiryquotation` 等 |
| **2. 原価回答** | 製造管理部が原価を登録。営業部がインポートし、利益を乗せて顧客用見積を作成。 | `s_estimatedata` → `t_quotation` |
| **3. 受注登録** | 見積データを受注・工事に変換。電子文書（見積書・注文書）を紐付け、OneDrive 等へ保存。 | `t_acceptorder` / `t_construction`<br>電子文書: `t_electricbooks` と紐付け |
| **4. 設計連携** | 設計部が Excel（パーツリスト）を作成。システムが読み込み、製作指示データを生成。 | Excel 取込 → `t_zumendata_manufct` 等 |

**データの流れ（受注・設計）:**

```
見積依頼(IQ) → 原価登録 → s_estimatedata → t_quotation
                                    ↓
受注登録 → t_acceptorder / t_construction ← 電子文書(t_electricbooks)
     ↓
設計 Excel(パーツリスト) → 取込 → t_zumendata_manufct 等
```

---

### ② 調達・製造フェーズ

| ステップ | 内容 | 主な保存先・連携 |
|----------|------|-------------------|
| **1. 発注** | 製造管理部が指示データをインポート。購入品（P番）・材料（R番）・外注加工（C番）を発注。 | 購入品(P番) → `t_purchaseparts`<br>材料(R番)/外注(C番) → `t_manufctpurchase` |
| **2. 納入** | 現場でスマホアプリでバーコード読み取り。納入日を更新。 | `t_purchaseparts` / `t_manufctpurchase` の **NounyuDate（納入日）** 更新 |
| **3. 社内加工・工数** | 作業者が「加工完了」をスマホで登録。作業票登録で工数実績を保存。 | 加工完了: 製作進捗テーブル<br>工数: `t_worktime`（機械単位） / `t_worktimekako` または `t_sagyofl`（図面単位） |

**データの流れ（調達・製造）:**

```
t_zumendata_manufct 等 → 発注
    ├─ P番 → t_purchaseparts
    └─ R番/C番 → t_manufctpurchase
           ↓
スマホ 納入処理 → NounyuDate 更新
           ↓
スマホ 加工完了登録 + 作業票登録 → t_worktime / t_worktimekako / t_sagyofl
```

---

### ③ 検収・売上フェーズ

| ステップ | 内容 | 主な保存先・連携 |
|----------|------|-------------------|
| **1. 検収（買掛）** | 届いた請求書に基づき「請求書アップロード」で金額照合。検収日を月末日として更新。 | `t_purchaseparts` 等の **KensyuDate（検収日）** を月末日で更新 |
| **2. 原価集計** | 夜間バッチが全実績を集計し、集計テーブルを生成。 | `t_工事番号別原価集計` 等 |
| **3. 出荷・売上（売掛）** | 工事番号出荷登録 → 請求日登録 → 売上登録。売上ステータスを更新。 | `t_moneyreceipt`、`t_acceptorder` の売上ステータス更新 |

**データの流れ（検収・売上）:**

```
請求書アップロード・照合 → t_purchaseparts 等 KensyuDate 更新（月末）
           ↓
夜間バッチ → t_工事番号別原価集計 等
           ↓
工事番号出荷登録 → 売上登録 → t_moneyreceipt / t_acceptorder 売上ステータス
```

---

| s_estimatedata / t_quotation | 原価・見積 | — | — |

## 3. 開発者向けの重要ロジック補足

実装時に必ず守るべき原則です。

### 3.1 主キーの原則

- **工事番号（ConstructionNo / constructno）は再利用される**ため、**常に「受注登録日（KeyDate）」との複合キー**でデータを特定するロジックが必要です。
- 検索・更新・削除・API の条件には `(constructno, keydate)` または `(constructno, registerdate)` を含める。
- 定数・ヘルパー: `js/erp-constants.js` の `getCompositeKey`, `buildCompositeKey`, `getKeyDate` を利用する。

### 3.2 電子帳簿保存法ロジック

- 注文書発行や売上登録の**トリガー**として、関連 PDF（見積・注文・請求）の**アップロードフラグをチェック**するバリデーションを設ける。
- アップロード未完了の場合は「発行不可」「売上登録不可」等とし、電子帳簿保存法に沿った運用とする。
- 紐付け先: `t_electricbooks` および OneDrive 等のストレージ。

### 3.3 ステータス管理

- **在庫（Stock）から手配（Arrange）への切り替え**、**取消（Cancel）と削除（Delete）の厳密な使い分け**など、フラグ管理が複雑です。
- 実装時は以下を明確に定義することを推奨します。
  - 各テーブルのステータス・フラグ一覧（例: CancelFlg, 手配済み, 納入済み, 検収済み）
  - 取消: 論理無効（フラグのみ。履歴は残す）
  - 削除: 物理削除または論理削除（deleted_at）のルール
- 画面・API では「取消済みは一覧に含めない／グレー表示」等の統一ルールを適用する。

## 4. モジュール間依存関係（設計時の参照）

```
経理・営業モジュール
  └─ 受注データ(t_acceptorder, t_construction) を参照・更新
       ↑
発注部門モジュール
  └─ 受注・設計データを参照し、t_purchaseparts / t_manufctpurchase を更新
  └─ 検収時に KensyuDate を更新
       ↑
原価・進捗モジュール
  └─ t_worktime / t_worktimekako 等を参照・登録
  └─ 夜間バッチ結果(t_工事番号別原価集計) を参照
       ↑
製造・現場（モバイル/PC）
  └─ t_purchaseparts / t_manufctpurchase の NounyuDate 更新
  └─ 製作進捗・工数テーブルを更新
```

- **共通キー**: すべてのモジュールで `(constructno, keydate)` を基準に紐付ける。
- **電子文書**: 経理・営業と発注部門の双方で、`t_electricbooks` およびアップロード完了フラグを参照する。
