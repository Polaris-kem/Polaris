# 生産管理システム 設計指針

神戸・明石の両工場の特性を活かし、現場が**「直感的に次のアクションを起こせる」**ことを重点に置いた設計指針です。  
プログラム構成と主要ロジックを定義し、単なるデータ入力ツールではなく「次の一手が分かる」仕組みを目指します。

**前提ドキュメント**: `docs/業務コンテキスト.md`, `docs/優先機能と方針.md`, `docs/画面と業務の対応.md`, `docs/用語・番号体系.md`

---

## 全体方針

| 観点 | 指針 |
|------|------|
| **設計思想** | 入力完了が目的ではなく、**一覧・色・アラートから「次に何をすべきか」が分かる**UI/ロジックを優先する。 |
| **拠点の役割** | 神戸＝組み立て・出荷の本流／明石＝単品加工・外注・毎日便で神戸へ。モジュールは拠点別ロジックを許容する。 |
| **データと紙** | 図面は紙＋検図が発注条件。システムは**検索・進捗・発注履歴・通知**を担い、紙運用と両立させる。 |
| **プログラム構成** | 受注・図面・外注購買・製造進捗・在庫余剰・UI/UX の6モジュールで整理し、共通レイヤー（通知・端数計算・検索）を共有する。 |

---

## 1. 受注・プロジェクト管理モジュール

長期プロジェクト（半年〜2年）の進捗と、頻繁な変更に対応するロジックを組み込む。

### 1.1 プログラム構成（想定）

```
受注・プロジェクト管理
├── 受注マスタ・CRUD（工事番号・希望納期・実際出荷日・金額・為替）
├── 分割請求・入金フラグ管理（契約時／中間／検証／完納）
├── 変更検知・通知トリガー（金額・納期・仕様変更時）
├── 見積/BOM連携（コピー・一括マージン加算）
└── 営業用入金確認ビュー（入金有無の参照専用）
```

### 1.2 主要ロジック

#### 1.2.1 分割請求ロジック

- **4段階入金フラグ**: 1受注あたり、次の4フェーズを論理的に管理する。
  - `payment_phase`: 契約時 | 中間 | 検証 | 完納（出荷）
- **データ構造案**: 受注テーブルに「契約時入金日・中間入金日・検証入金日・完納入金日」の4カラム、または入金明細テーブルで `phase` をキーに管理。総務の消し込みは基幹で実施し、当システムは**営業が「入金されたか」を見る専用画面**から参照する。
- **表示**: 一覧・詳細で「次の請求フェーズ」「未入金フェーズ」を強調し、**次アクション（どのフェーズの請求・確認か）**が一目で分かるようにする。

#### 1.2.2 変更履歴・通知機能

- **トリガー対象**: 受注金額・希望納期・実際出荷日・仕様メモ等の変更。
- **実装方針**:
  - 更新前後の値を比較し、変更があった場合にのみ「変更通知キュー」へ投入（または即時送信）。
  - 通知先は受注に紐づく担当（営業・製造管理・資材等）をマスタまたは受注データから解決し、**メール／社内通知**を送る。
- **履歴**: 変更日時・項目・旧値・新値・実行者を履歴テーブルに保存し、一覧または詳細から「変更履歴」を開けるようにする。長期案件の経緯追跡に利用。

#### 1.2.3 見積・BOM連携（一括マージン加算）

- **用途**: 過去の見積・BOMをコピーして新規受注に流用する際、材料高騰分を反映する。
- **ロジック**:
  - コピー元（工事番号・見積版）を指定し、**品目単位または合計に対してマージン率／マージン額を一括適用**する。
  - 計算結果をプレビュー表示し、確定後に受注・BOMに反映。必要なら「一式（000番）」単位の簡略構成にも対応する。
- **UI**: 「見積コピー＆マージン加算」のような単一フローにし、**コピー → 率入力 → プレビュー → 確定**の順で直感的に完了できるようにする。

## 2. 図面・技術管理モジュール

業務の起点となる「図面」をデータと物理（紙）の両面で管理する。

### 2.1 プログラム構成（想定）

```
図面・技術管理
├── 図面番号採番・登録（部品図／組立図）
├── 図面履歴検索エンジン（番号・名称・期間）
├── 訂正管理（A・B・C訂正、発注番号維持）
└── 検図状態・紙図面連携（発注条件フラグ）
```

### 2.2 主要ロジック

#### 2.2.1 図面履歴検索エンジン

- **検索方式**:
  - **完全一致**: 図面番号で検索。インデックスを効かせて高速化。
  - **部分一致・曖昧検索**: 図面番号の前方一致、含む検索。名称（品名・部品名）での検索も必須。
- **期間フィルタ**: **過去3年分に遡る**をデフォルトまたは1クリックで選択可能にし、結果セットを日付範囲で限定。パフォーマンス対策として、集計済みビューやキャッシュを検討する。
- **表示**: 検索結果一覧では、**色分け（設計中=水色、緊急=赤、完了=紫）**と「次のアクション」（発注待ち・納入待ち等）を併せて表示し、何をすべきかが分かるようにする。

#### 2.2.2 訂正管理（A・B・C訂正）

- **要件**: 図面の訂正連絡書に基づき、**発注済みデータに「訂正あり」フラグを立てるが、発注番号は変更しない**。
- **ロジック**:
  - 図面番号＋訂正版（A/B/C等）をバージョンとして管理。発注・手配データには「図面版」「訂正版」を保持。
  - 訂正登録時は、既存発注行に対して「訂正版を更新」し、**発注番号・枝番は維持**。履歴上は「訂正前→訂正後」が追えるようにする。
- **画面**: 図面詳細または発注一覧で「訂正あり」をバッジ／色で表示し、現場が再手配・再発注の要否を判断しやすくする。

## 3. 外注購買・資材管理モジュール

製造管理部と資材課の異なる発注フローを統合する。

### 3.1 プログラム構成（想定）

```
外注購買・資材管理
├── 依頼書・注文書の2段階ワークフロー
├── 月末バッチ（依頼書→注文書集計・確定）
├── 再発注・送料追加（枝番維持）
├── 端数計算マスタ（取引先別 四捨五入/切捨/切上）
└── 請求照合時の自動端数適用
```

### 3.2 主要ロジック

#### 3.2.1 2段階発注ワークフロー

- **第1段階（依頼書）**: 現場が発行する「依頼書」を**未検収の正式発注**として登録。発注日・業者・明細・納期を確定させる。
- **第2段階（注文書）**: 月末に、当月分の依頼書を集計し、**注文書として確定するバッチ処理**を実行。請求・支払いの単位とする。
- **データ構造**: 依頼書テーブル（発注単位）と注文書テーブル（集計単位）をリレーション。依頼書には「注文書確定済み」フラグを持たせ、バッチ実行時に更新する。
- **UI**: 「今月分の依頼を注文書に反映」のような1アクションで実行できるようにし、実行結果（件数・金額サマリ）を表示して**次のアクション（照合・支払い）**に繋げる。

#### 3.2.2 再発注・送料追加処理

- **再発注**: 破損等で「1個だけ追加」する場合、**既存の構成（親番号・枝番）は変更せず**、再発注用の明細または別テーブルで「同一品目の追加発注」として記録。履歴で「本発注＋再発注」が追えるようにする。
- **送料追加**: 見積確定後に送料が判明した場合、**対象明細の右クリック等で「送料追加」**を実行。元の枝番を維持したまま、送料行を同一発注に差し込む。金額・請求照合時に送料が含まれるようにする。

#### 3.2.3 端数計算マスタ

- **取引先マスタ**: 業者ごとに「端数計算方法」を保持。  
  - 候補: `round`（四捨五入）, `floor`（切り捨て）, `ceil`（切り上げ）の3パターン。
- **請求照合時**: 発注金額・税込計算時に、その業者の端数ロジックを自動適用し、**1円単位の誤差が出ない**ようにする。照合画面では「システム計算額」と「請求書額」を並べて表示し、不一致時のみ要確認とできるようにする。

## 4. 製造・進捗管理モジュール（工場別ロジック）

神戸と明石の特性に合わせた拠点別ロジックを許容する。

### 4.1 プログラム構成（想定）

```
製造・進捗管理
├── 拠点間物流（毎日便）・現品票突合
├── リン酸マンガン処理（1括発注 : N個体バーコード）
├── 進捗ステータス・色分けエンジン
└── カレンダー／リストビュー表示
```

### 4.2 主要ロジック

#### 4.2.1 拠点間物流（毎日便）管理

- **流れ**: 明石で受入した外注品・加工品を、毎日便で神戸へ転送。神戸側で「何が届いたか」を現品と突合する。
- **突合ロジック**:
  - 明石受入時に**現品票（バーコード）**を発行・登録。毎日便に載せる明細とバーコードを紐付ける。
  - 神戸到着後、**バーコードスキャンまたは明細選択**で「受入済み」を登録。未突合リストを表示し、**次にスキャンすべき品**が分かるようにする。
- **画面**: 「本日の毎日便 未受入一覧」「バーコードスキャン受入」など、**今日やるべき受入**が一目で分かるUIにする。

#### 4.2.2 リン酸マンガン処理（1:N管理）

- **特性**: 「1括（1盛り）発注」で複数部品をまとめて発注するが、**どの部品が処理済みかは個別に管理**する必要がある。
- **データ構造**:
  - 親: 1回の「括り発注」（発注番号・業者・発注日）。
  - 子: N個の部品。各子に**固有バーコード**と「加工完了フラグ」「完了日」を持たせる。
- **ロジック**: 戻り仕分け後、バーコード読み取りで**個別に「加工完了」を登録**。親発注の進捗は「子の完了数 / N」で算出し、一覧で「未完了・完了」を色分けする。発注番号は維持したまま、個体単位の進捗だけ更新する。

#### 4.2.3 視覚的ステータス管理

- **色の定義**（`docs/用語・番号体系.md` と統一）:
  - **水色**: 設計データ（設計から上がった状態）
  - **赤**: 緊急
  - **黄**: 未完了（進行中）※既存運用と合わせて必要なら追加
  - **紫**: 完成（入力済み）
- **適用箇所**: リストビュー（受注一覧・図面一覧・加工進捗一覧）、カレンダー（納入予定・出荷予定）。ステータスまたは期日から**動的にクラスを付与**し、色で次のアクションが分かるようにする。
- **実装**: ステータス値→色のマッピングを共通定数または設定に持ち、一覧描画時に適用する。

## 5. 在庫・余剰品活用モジュール

「在庫があるのに発注してしまう」ミスを防ぐ。

### 5.1 プログラム構成（想定）

```
在庫・余剰品活用
├── 余剰在庫マスタ（登録・検索）
├── 購買依頼/BOM取り込み時の自動照合
├── 在庫ありアラート・ポップアップ
└── 一覧の「在庫あり」列・色強調
```

### 5.2 主要ロジック

#### 5.2.1 自動在庫引き当て・アラート

- **トリガー**: 購買依頼登録時、またはBOM取り込みで発注候補が並んだ時点で、**余剰在庫マスタを検索**（図面番号・品番・品名等で照合）。
- **一致時**:
  - **ポップアップ**: 「同一品目が余剰在庫に○件あります」と表示。在庫詳細へ遷移できるリンクを付ける。
  - **一覧での強調**: 発注一覧の該当行に「在庫あり」列を表示、または行を色分けして**発注前に気づける**ようにする。
- **運用**: 在庫を使用する場合は、余剰レコードと紐付けて減数を更新する処理を将来拡張できるようにしておく（優先度は中〜低）。

## 6. UI/UX・運用支援の要件

現場が「システムを怖がらずに触れる」ための配慮。

### 6.1 プログラム構成（想定）

```
UI/UX・運用支援
├── 検索条件の保存・読込（ユーザー別）
├── ダッシュボード・クイックアクション（次のアクション誘導）
└── エラーメッセージ・確認ダイアログの統一
```

### 6.2 主要ロジック

#### 6.2.1 検索条件の保存機能

- **内容**: 一覧画面の検索条件（工事番号・日付範囲・ステータス・担当等）を**名前を付けて保存**し、ユーザーごとに保持する。
- **データ**: ユーザーID・画面ID・保存名・条件（JSON等）を「保存済み検索」テーブルに格納。
- **UI**: 「現在の条件を保存」「保存した条件を呼び出し」を1クリックで実行。**よく使う抽出を登録しておき、ワンクリックで同じ一覧を再表示**できるようにする。次のアクション（例: 未完了だけ）にすぐ到達できる。

#### 6.2.2 その他

- **ダッシュボード**: 「本日の毎日便未受入」「未入金フェーズあり」「緊急（赤）の件数」など、**今やるべきこと**を少数に絞って表示する。
- **確認・エラー**: 削除・確定など取り消しにくい操作は確認ダイアログで統一。エラーは「何が原因か」「どう直すか」が分かる短文で表示する。

## 7. 共通レイヤー・横断事項

| 項目 | 内容 |
|------|------|
| **通知** | 変更通知・見積OK通知などは、キューまたは即時送信の共通サービスに集約し、各モジュールから呼び出す。 |
| **端数計算** | 取引先マスタの設定を参照する共通関数を用意し、購買・請求の両方で利用する。 |
| **色・ステータス** | ステータス→色のマッピングは共通定数／設定で一元管理し、受注・図面・製造の各一覧で同じルールを適用する。 |
| **権限** | 営業／資材／製造管理／総務など、参照・更新できる画面・テーブルを役割で制御する方針を将来定義する。 |

## 8. 実装状況（プログラムへの反映）

- **共通ロジック**: `js/polaris-production.js` に 6 モジュールの定数・API を実装（分割請求、ステータス色、端数計算、余剰照合、検索条件保存など）。
- **受注一覧**: 「次の請求」列の表示、行のステータス色（水色/赤/黄/紫）を適用。`loadAcceptOrderList` で `PolarisProduction.getPaymentPhaseSummary` / `getStatusStyle` を利用。
- **汎用一覧**: テーブル表示時に `PolarisProduction.getStatusStyle` で行の色を付与。
- **検索条件の保存**: 一覧画面に「条件を保存」「保存した条件で復元」を追加。`localStorage` に保存し、テーブル名・検索語を復元。
- **余剰在庫チェック**: 一覧画面の「余剰在庫をチェック」で表示行と `t_surplus` を照合し、一致行を緑で強調（`.polaris-surplus-match`）。

DB に「契約時入金日」「中間入金日」等のカラムが存在する場合、受注一覧の「次の請求」が自動で反映されます。未定義の場合は「契約時」を次のフェーズとして表示します。
